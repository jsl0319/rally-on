# RallyOn — 디자인 시스템 명세

> 테린이가 모임의 기대치와 코트 상태를 빠르게 이해하도록 하는 UI 표현 기준. 마지막 갱신: 2026-07-14

## 1. 목적과 적용 범위

이 문서는 [프로젝트 브리프](../PROJECT_BRIEF.md)의 MVP 화면을 일관되게 구현하기 위한 디자인 토큰, 컴포넌트, 콘텐츠, 화면 상태의 기준이다.

RallyOn의 UI는 사람을 서열화하거나 친목을 강요하지 않는다. 사용자가 신청 전에 **무엇을·누구와·어디서·어떤 규칙으로** 운동하는지 판단하도록 돕는다.

## 2. 디자인 원칙

| 원칙 | 구현 기준 |
| --- | --- |
| 실제 플레이 우선 | 신청 수가 아닌 출석 완료를 성공 상태로 표현한다. |
| 기대치의 가시화 | 목적·레벨·운동 계획·코트·비용·우천 규칙을 신청 전 노출한다. |
| 코트 상태의 분리 | `코트 예약 완료`와 `코트 찾는 중`을 같은 시각적 무게로 표현하지 않는다. |
| 비경쟁적 언어 | 고수/하수, 순위, 승률을 사용하지 않고 실제 플레이 경험을 쓴다. |
| 안전한 신뢰 | 본인인증·승인·재확인·출석·구조화된 후기를 연결한다. |
| 이미지의 절제 | 장소 판단에 도움이 되는 코트 이미지만 사용하고, 인물 평가나 과도한 친목 연출에는 쓰지 않는다. |

## 3. 디자인 토큰

### 3.1 색상

| 토큰 | 값 | 용도 |
| --- | --- | --- |
| `color.brand` | `#176B4D` | 주요 CTA, 선택 상태, 확정 행동 |
| `color.brand-weak` | `#E7F3EC` | 선택된 필터, 긍정 상태 배경 |
| `color.ink` | `#1B1F1C` | 제목·본문 |
| `color.muted` | `#66706A` | 보조 정보·메타데이터 |
| `color.surface` | `#FFFFFF` | 기본 표면 |
| `color.canvas` | `#F7F8F6` | 앱 배경 |
| `color.line` | `#D9DED9` | 구분선·비활성 테두리 |
| `color.warning` | `#9A5A00` | 코트 미확보·주의 안내 |
| `color.warning-weak` | `#FFF3D8` | 주의 상태 배경 |
| `color.danger` | `#B42318` | 취소·신고·오류 |
| `color.danger-weak` | `#FDECEC` | 위험 상태 배경 |

- 본문과 배경의 대비는 WCAG AA(일반 텍스트 4.5:1 이상)를 만족한다.
- 색상만으로 상태를 전달하지 않는다. 모든 상태에는 텍스트 라벨과 아이콘 또는 형태 차이를 함께 둔다.
- `코트 찾는 중`은 경고색을 쓰되 실패·오류처럼 보이게 하지 않는다.

### 3.2 타이포그래피·간격·모서리

| 구분 | 토큰 | 기준 |
| --- | --- | --- |
| 글꼴 | `font.sans` | `Pretendard`, `Apple SD Gothic Neo`, `system-ui`, sans-serif |
| 제목 | `type.title` | 22px / 700 / 1.35 |
| 섹션 제목 | `type.heading` | 18px / 700 / 1.4 |
| 본문 | `type.body` | 15px / 400 / 1.55 |
| 보조 | `type.meta` | 13px / 400 / 1.45 |
| 간격 단위 | `space.1` ~ `space.8` | 4, 8, 12, 16, 20, 24, 32, 40px |
| 모서리 | `radius.sm/md/lg` | 8, 12, 16px |
| 터치 영역 | `size.touch` | 최소 44 × 44px |

### 3.3 그림자·이미지

- 카드에는 기본적으로 테두리만 사용하고, 떠 있는 모달·하단 시트에만 약한 그림자를 쓴다.
- 코트 사진은 `16:9` 또는 `16:8` 비율로 자르고, `object-fit: cover`를 사용한다.
- 장소 사진에는 코트·실내외·조명 등 참여 판단에 유용한 정보를 담는다.
- 탐색·상세 화면에서 사용하는 장소 이미지 원본은 [`assets/images/seoul-tennis-court.jpg`](../assets/images/seoul-tennis-court.jpg)이다.

## 4. 핵심 컴포넌트

### 4.1 버튼

| 변형 | 사용 시점 | 예시 |
| --- | --- | --- |
| Primary | 현재 화면의 단일 핵심 행동 | 참여 신청하기, 참석 준비 완료, 후기 완료 |
| Secondary | 보조 행동 | 모임 자세히 보기, 모임 보기 |
| Text | 뒤로가기, 신고, 낮은 중요도 이동 | `‹ 탐색`, `문제가 있었어요` |
| Destructive | 되돌리기 어려운 행동의 최종 확인 | 모임 취소 확정, 차단 확정 |

- 한 화면에 Primary 버튼은 하나만 둔다.
- 비활성 버튼은 이유가 화면에 드러날 때만 사용한다. 예: `코트 정보를 입력하면 공개할 수 있어요.`
- 신청·승인·취소처럼 서버 상태를 바꾸는 버튼은 요청 중 중복 클릭을 막고 `처리 중` 상태를 보여준다.

### 4.2 상태 배지

| 배지 | 데이터 값 | 표현 | 사용 위치 |
| --- | --- | --- | --- |
| 코트 예약 완료 | `courtStatus = secured` | 브랜드 약한 배경 + `코트 예약 완료` | 모임 카드·상세·채팅 공지 |
| 코트 찾는 중 | `courtStatus = needed` | 경고 약한 배경 + `코트 찾는 중` | 대기 매칭 카드·상세 |
| 모집 중 | `meetingStatus = recruiting` | 중립 배지 | 호스트 화면·상세 |
| 승인 대기 | `participantStatus = pending` | 중립 배지 | 신청자·내 모임 |
| 참석 확인 필요 | `meetingStatus = reconfirming` | 경고 배지 + 마감 시각 | 내 모임·알림 |
| 취소됨 | `meetingStatus = cancelled` | 위험 약한 배경 + 사유 | 내 모임·알림 |

상태값의 허용 전환과 데이터 의미는 [데이터 상태값 명세](data-state-spec.md)를 따른다.

### 4.3 모임 카드

모임 카드의 정보 우선순위는 아래와 같다.

1. 코트 확보 여부, 날짜·시간
2. 운동 목적과 실제 플레이 경험 기준
3. 코트명·실내외·운동 시간
4. 현재 인원/정원, 1인 비용
5. 호스트 신뢰 요약, 첫 참여 환영 여부

코트 사진은 카드의 첫 모임 또는 상세 화면에서 사용한다. 이미지가 없을 때는 지도·실시간 위치를 대신 노출하지 않고, 코트명과 실내외 정보를 유지한다.

### 4.4 구조화된 정보 블록

`일시`, `코트`, `비용`, `인원`, `우천 규칙`처럼 신청 판단에 중요한 정보를 레이블-값 형식으로 표시한다.

- 값이 없으면 빈칸 대신 상태를 쓴다. 예: `코트: 찾는 중`, `우천 규칙: 호스트 확인 필요`.
- 금액은 `1인 예상 비용`과 `정산 방식`을 함께 표시한다.
- 실제 거주지, 정밀 위치, 전화번호는 표시하지 않는다.

### 4.5 피드백·안전 컴포넌트

- 후기는 이진 또는 제한된 선택지로만 제공한다. 자유로운 외모·신체 평가 입력란을 두지 않는다.
- 신고는 별도 확인 시트로 열고, 신고 대상·사유·증거 설명을 수집한다.
- 차단은 확인 후 즉시 적용하며, 차단한 사용자에게는 이후 탐색·신청·채팅에서 상대를 노출하지 않는다.

## 5. 공통 화면 상태

각 데이터 의존 화면은 반드시 아래 상태를 구현한다.

| 상태 | UI 기준 |
| --- | --- |
| Loading | 화면 구조와 같은 스켈레톤을 표시한다. |
| Empty | 이유와 다음 행동을 함께 제공한다. 예: `조건에 맞는 코트 확보 모임이 없어요. 코트 찾는 중 모임도 볼까요?` |
| Error | 기술 오류를 노출하지 않고 재시도와 안전한 이전 행동을 제공한다. |
| Permission denied | 필요한 권한과 거부해도 가능한 대안을 설명한다. 위치 권한 거부 시 선호 지역 직접 선택을 제공한다. |
| Offline/stale | 마지막 갱신 시각을 보여주고, 신청·승인 같은 변경 행동은 막는다. |
| Success | 상태가 바뀐 이유와 다음 행동을 명확히 표시한다. 예: `신청 완료 · 호스트 승인 후 채팅이 열려요.` |

## 6. 화면별 점검 기준

| 화면 | 반드시 보여야 하는 것 |
| --- | --- |
| 온보딩 | 지역, 실제 플레이 경험, 오늘의 목적, 참여 선호 |
| 모임 탐색 | 필터, 코트 상태, 목적·레벨, 시간·장소·비용·인원 |
| 모임 상세 | 운동 계획, 우천 규칙, 신청 상태, 승인 안내 |
| 모임 만들기 | 코트 확보/미확보의 분리, 템플릿, 공개 전 필수값 |
| 승인·채팅 | 목적·레벨 적합도, 고정 공지, 참여자 상태 |
| 재확인·출석 | 코트·집결·비용·우천, 참석/출석 확인 |
| 후기 | 시간·레벨·배려·재매칭의 제한된 문항, 신고 진입점 |
